## 1. 引言

### 1.1 编写目的

本文档用于说明“珞珈线上点餐系统”的软件需求，明确系统应实现的功能范围、用户角色、数据模型、接口要求、界面要求、非功能性要求、团队分工与验收标准。

本文档可作为以下工作的依据：

- 项目开发依据
- 小组成员分工依据
- 接口联调依据
- 测试与验收依据
- 课程大作业提交材料依据

### 1.2 项目背景

随着餐饮行业信息化水平不断提高，传统人工点餐方式存在效率低、出错率高、订单状态不透明、经营数据难统计等问题。本项目拟开发一套基于 C# 技术栈的线上点餐系统，实现顾客端自助点餐、管理端订单处理、后端 API 服务、MySQL 数据存储以及 AI 推荐/经营建议等功能。

本项目为 C# 课程大作业，由 3 人团队在 2 周内完成，采用“后端 WebAPI + 两个 WinForms 客户端 + MySQL 数据库”的架构。

## 2. 项目概述

### 2.1 系统目标

本系统目标是实现一个面向餐厅场景的线上点餐系统，支持顾客自助浏览菜单、添加购物车、提交订单，支持管理员进行菜品维护、订单处理、销售统计和 AI 经营建议查看。

系统应覆盖以下核心目标：

1. 顾客能够通过 WinForms 顾客端完成点餐流程。
2. 管理员能够通过 WinForms 管理端维护菜品并处理订单。
3. 后端通过 ASP.NET Core WebAPI 提供统一数据接口。
4. 数据通过 MySQL 持久化存储。
5. 系统支持热销菜品统计和 AI 推荐/经营建议功能。
6. 系统能够满足课程大作业演示、测试和文档提交要求。

### 2.2 系统范围

系统由三部分组成：

```text

│   ├── 购物车管理
│   ├── 堂食/外卖信息填写
│   ├── 提交订单

│   ├── 菜品列表查看
```

## 3. 系统业务流程分析

本章从业务流程角度说明系统在实际使用过程中的运行方式，重点描述顾客点餐、管理员处理订单、菜品维护、统计分析以及 AI 推荐等主要业务流程。该部分用于帮助开发人员和测试人员理解系统的整体业务闭环，并为后续功能设计和验收测试提供依据。

### 3.1 顾客点餐业务流程

顾客点餐是本系统的核心业务流程。顾客进入系统后，可以选择堂食或外卖点餐模式，然后浏览菜品、加入购物车并提交订单。系统在接收到订单后，将订单信息保存到数据库，并等待管理员处理。

顾客点餐流程如下：

1. 顾客打开顾客端程序；
2. 顾客选择堂食或外卖点餐模式；
3. 若选择堂食，顾客填写桌号；若选择外卖，顾客填写配送地址和联系方式；
4. 顾客浏览菜品列表，也可以通过分类筛选或关键字搜索查找菜品；
5. 顾客查看菜品名称、价格、分类、图片、辣度和描述等信息；
6. 顾客选择需要的菜品并加入购物车；
7. 顾客在购物车中修改菜品数量或删除不需要的菜品；
8. 系统自动计算菜品小计、配送费和订单总金额；
9. 顾客确认订单内容并填写备注信息；
10. 顾客提交订单；
11. 后端接收订单请求，将订单主表和订单明细保存到 MySQL 数据库；
12. 系统生成订单编号，并将订单初始状态设置为 `Pending`；
13. 顾客可根据订单编号查询订单状态。

### 3.2 管理员处理订单业务流程

管理员主要负责查看顾客提交的订单，并根据餐厅实际处理进度修改订单状态。订单状态用于反映订单从待处理到完成的整个过程，便于顾客了解当前订单进度。

管理员处理订单流程如下：

1. 管理员打开管理端程序；
2. 管理员进入订单管理页面；
3. 系统从后端获取订单列表并显示订单基本信息；
4. 管理员选择某一订单查看订单详情；
5. 管理员根据实际处理情况修改订单状态；
6. 后端接收状态修改请求，并更新数据库中的订单状态；
7. 管理端刷新订单列表，显示最新订单状态；
8. 顾客端查询订单时，可获取更新后的订单状态。

订单状态变化建议如下：

1. `Pending`：订单已提交，等待管理员处理；
2. `Confirmed`：管理员已确认订单；
3. `Preparing`：订单正在制作中；
4. `Ready`：菜品已制作完成，堂食订单可取餐，外卖订单等待配送；
5. `Delivering`：外卖订单正在配送中；
6. `Completed`：订单已完成；
7. `Cancelled`：订单已取消。

### 3.3 菜品管理业务流程

菜品管理是管理员端的重要功能。管理员可以对菜品信息进行维护，包括新增菜品、修改菜品、删除菜品或设置菜品为不可售。顾客端展示的菜品信息来自后端数据库，因此管理员修改菜品后，顾客端刷新即可获取最新菜品数据。

菜品管理流程如下：

1. 管理员打开管理端程序；
2. 管理员进入菜品管理页面；
3. 系统调用菜品接口，从数据库中读取菜品列表；
4. 管理员查看当前菜品信息；
5. 管理员根据需要新增、修改或删除菜品；
6. 管理员填写或修改菜品名称、分类、价格、图片、辣度、描述和是否可售等信息；
7. 管理端将菜品信息提交到后端接口；
8. 后端校验菜品数据是否合法；
9. 后端通过 EF Core 更新 MySQL 数据库；
10. 管理端提示操作结果；
11. 顾客端刷新后显示最新菜品信息。

### 3.4 统计分析业务流程

系统根据订单和订单明细数据统计菜品销量、销售金额等信息，管理员可以在管理端查看热销菜品排行和营收情况，为餐厅备料和经营决策提供参考。

统计分析流程如下：

1. 顾客提交订单后，系统保存订单数据和订单明细数据；
2. 管理员进入统计分析页面；
3. 管理端向后端统计接口发送请求；
4. 后端根据订单明细汇总菜品销量和销售额；
5. 后端按照销量或销售额对菜品进行排序；
6. 后端返回热销菜品排行和营收统计结果；
7. 管理端以表格或图表形式展示统计数据；
8. 管理员根据统计结果了解菜品销售情况。

### 3.5 AI 推荐与经营建议业务流程

AI 功能主要包括顾客端菜品推荐和管理端经营建议。顾客端可以根据当前购物车内容或口味偏好获取推荐菜品；管理端可以根据销售数据获取经营建议。若 AI 服务调用失败，系统应返回本地模拟推荐或模板化建议，保证系统演示流程正常进行。

顾客端 AI 菜品推荐流程如下：

1. 顾客在点餐过程中选择部分菜品；
2. 顾客点击获取 AI 推荐；
3. 顾客端将当前已选菜品和口味偏好发送给后端；
4. 后端整理请求数据并调用 AI 推荐服务；
5. AI 服务返回推荐菜品和推荐理由；
6. 后端将推荐结果返回顾客端；
7. 顾客端在推荐区域展示推荐内容。

管理端 AI 经营建议流程如下：

1. 管理员进入 AI 建议页面；
2. 管理端请求后端获取销售统计数据；
3. 后端整理热销菜品、低销量菜品和订单营收等信息；
4. 后端调用 AI 服务生成经营建议；
5. AI 服务返回建议文本；
6. 管理端展示经营建议；
7. 管理员可根据建议调整备料、菜单或促销策略。

### 3.6 系统整体业务闭环

本系统的整体业务闭环可以概括为“顾客下单、后端存储、管理员处理、状态反馈、数据统计、辅助决策”。顾客端负责产生订单，后端负责保存和处理数据，管理端负责订单处理和经营管理，统计与 AI 功能则用于提升系统的实用性和展示效果。

系统整体业务闭环如下：

1. 顾客在顾客端浏览菜品并提交订单；
2. 后端接收订单并保存到数据库；
3. 管理员在管理端查看和处理订单；
4. 后端更新订单状态；
5. 顾客端查询并显示最新订单状态；
6. 系统根据历史订单生成统计数据；
7. AI 服务根据点餐数据或销售数据生成推荐和建议；
8. 管理员根据统计分析和 AI 建议优化餐厅经营。

## 4. 用户角色与使用场景

### 4.1 用户角色

| 角色   | 使用端          | 主要权限                         |
| ---- | ------------ | ---------------------------- |
| 顾客   | 顾客端 WinForms | 浏览菜品、加入购物车、提交订单、查询订单状态、获取推荐  |
| 管理员  | 管理端 WinForms | 管理菜品、查看订单、修改订单状态、查看统计、获取经营建议 |

### 4.2 典型使用场景

#### 4.2.1 顾客堂食点餐

1. 顾客打开顾客端。
2. 选择堂食点餐模式。
3. 输入桌号。
4. 浏览或搜索菜品。
5. 将菜品加入购物车。
6. 修改数量或删除购物车项目。
7. 填写备注。
8. 提交订单。
9. 系统返回订单号和初始状态。
10. 顾客可查询订单状态。

#### 4.2.2 顾客外卖点餐

1. 顾客打开顾客端。
2. 选择外卖点餐模式。
3. 输入姓名、电话和配送地址。
4. 浏览菜品并加入购物车。
5. 系统计算商品金额和配送费。
6. 顾客提交订单。
7. 系统保存订单并显示状态。
8. 管理员处理订单后，顾客可查看状态变化。

#### 4.2.3 管理员维护菜品

1. 管理员打开管理端。
2. 进入菜品管理页面。
3. 查看当前菜品列表。
4. 添加新菜品或修改已有菜品。
5. 设置菜品分类、价格、辣度、图片、是否可售等信息。
6. 保存后，顾客端刷新可看到最新菜品。

#### 4.2.4 管理员处理订单

1. 管理员进入订单管理页面。
2. 查看订单列表和订单详情。
3. 根据厨房处理进度修改订单状态。
4. 订单状态可从 `Pending` 更新为 `Confirmed`、`Preparing`、`Ready`、`Delivering`、`Completed` 或 `Cancelled`。
5. 顾客端可查询到最新订单状态。

#### 4.2.5 管理员查看统计与 AI 建议

1. 管理员进入统计分析页面。
2. 系统调用统计接口获取热销菜品排行。
3. 管理端使用图表展示统计结果。
4. 管理员进入 AI 建议页面。
5. 系统根据销售数据生成经营建议。
6. 管理员根据建议进行备料或菜单调整。

## 5. 总体需求

### 5.1 业务需求

| 编号     | 需求名称    | 需求描述                     | 优先级 |
| ------ | ------- | ------------------------ | --- |
| BR-001 | 顾客自助点餐  | 顾客能够在客户端自主完成菜品浏览、选择和下单   | 高   |
| BR-002 | 管理员处理订单 | 管理员能够查看订单并更新订单状态         | 高   |
| BR-003 | 菜品后台维护  | 管理员能够对菜品进行增删改查           | 高   |
| BR-004 | 数据持久化   | 菜品、订单、订单明细应保存到 MySQL 数据库 | 高   |
| BR-005 | 热销统计    | 系统能够统计销量靠前的菜品            | 中   |
| BR-006 | AI 推荐   | 系统能够为顾客提供菜品推荐            | 中   |
| BR-007 | AI 经营建议 | 系统能够为管理员提供经营建议           | 中   |
| BR-008 | 演示可用    | 系统应能够支持课堂演示和验收           | 高   |

### 5.2 系统架构需求

系统采用 C/S + WebAPI 架构：

```text
ClientCustomer / ClientAdmin
        |
        | HTTP JSON
        v
OnlineOrdering.API
        |
        | EF Core
        v
MySQL Database
```

要求如下：

1. 顾客端和管理端均通过 HTTP 请求访问后端。
2. 后端统一处理业务逻辑和数据访问。
3. 数据库只由后端访问，客户端不得直接连接数据库。
4. 接口数据格式统一使用 JSON。
5. 后端应支持 Swagger，便于接口调试和联调。

## 7. 数据需求

### 7.1 菜品数据

#### 7.1.1 当前字段

当前基础模型包含：

| 字段         | 说明    |
| ---------- | ----- |
| `Id`       | 菜品 ID |
| `Name`     | 菜品名称  |
| `Category` | 菜品分类  |
| `Price`    | 菜品价格  |

#### 7.1.2 需扩展字段

| 字段            | 类型     | 说明       | 是否必需 |
| ------------- | ------ | -------- | ---- |
| `ImagePath`   | string | 菜品图片路径   | 否    |
| `SpicyLevel`  | int    | 辣度等级，0-3 | 否    |
| `IsAvailable` | bool   | 是否可售     | 是    |
| `Description` | string | 菜品描述     | 否    |

#### 7.1.3 推荐 Dish 实体

```csharp
public class Dish
{
    public int Id { get; set; }

    public string Name { get; set; } = string.Empty;

    public string Category { get; set; } = string.Empty;

    public decimal Price { get; set; }

    public string? ImagePath { get; set; }

    public int SpicyLevel { get; set; } = 0;

    public bool IsAvailable { get; set; } = true;

    public string? Description { get; set; }
}
```

### 7.2 订单数据

#### 7.2.1 当前字段

当前基础模型包含：

| 字段             | 说明    |
| -------------- | ----- |
| `Id`           | 订单 ID |
| `CustomerName` | 顾客姓名  |
| `Phone`        | 联系电话  |
| `Address`      | 地址    |
| `TotalAmount`  | 总金额   |
| `Status`       | 订单状态  |
| `CreatedAt`    | 创建时间  |
| `OrderItems`   | 订单明细  |

#### 7.2.2 需扩展字段

| 字段            | 类型      | 说明                    | 是否必需 |
| ------------- | ------- | --------------------- | ---- |
| `TableNumber` | string  | 桌号，堂食时使用              | 堂食必填 |
| `Note`        | string  | 订单备注                  | 否    |
| `OrderType`   | string  | `DineIn` 或 `Delivery` | 是    |
| `DeliveryFee` | decimal | 配送费                   | 外卖需要 |

#### 7.2.3 推荐 Order 实体

```csharp
public class Order
{
    public int Id { get; set; }

    public string CustomerName { get; set; } = string.Empty;

    public string Phone { get; set; } = string.Empty;

    public string? Address { get; set; }

    public string? TableNumber { get; set; }

    public string? Note { get; set; }

    public decimal TotalAmount { get; set; }

    public decimal DeliveryFee { get; set; } = 0.00m;

    public string OrderType { get; set; } = "DineIn";

    public string Status { get; set; } = "Pending";

    public DateTime CreatedAt { get; set; } = DateTime.Now;

    public List<OrderItem> OrderItems { get; set; } = new();
}
```

### 7.3 订单明细数据

| 字段         | 类型      | 说明      |
| ---------- | ------- | ------- |
| `Id`       | int     | 明细 ID   |
| `OrderId`  | int     | 所属订单 ID |
| `DishId`   | int     | 菜品 ID   |
| `DishName` | string  | 下单时菜品名称 |
| `Price`    | decimal | 下单时菜品单价 |
| `Quantity` | int     | 数量      |

### 7.4 数据关系

```text
Dish 1 --- N OrderItem N --- 1 Order
```

说明：

- 一个菜品可以出现在多个订单明细中。
- 一个订单包含多个订单明细。
- 订单明细保存菜品名称和下单价格，避免菜品后续改价影响历史订单。

## 10. 非功能性需求

### 10.1 性能需求

| 编号      | 需求                       |
| ------- | ------------------------ |
| NFR-001 | 普通 API 请求响应时间应控制在 2 秒以内。 |
| NFR-002 | 顾客端加载菜品列表时不应造成界面长时间卡顿。   |
| NFR-003 | 管理端刷新订单列表时应支持异步加载。       |
| NFR-004 | 统计接口在测试数据规模下应能快速返回结果。    |

### 10.2 易用性需求

| 编号      | 需求                     |
| ------- | ---------------------- |
| NFR-005 | 顾客端界面应简洁直观，按钮含义明确。     |
| NFR-006 | 管理端应便于快速查看订单和修改状态。     |
| NFR-007 | 错误信息应使用中文提示，不直接暴露异常堆栈。 |
| NFR-008 | 必填项为空时应给出明确提示。         |

### 10.3 可靠性需求

| 编号      | 需求                               |
| ------- | -------------------------------- |
| NFR-009 | 后端应对数据库连接失败、参数错误、资源不存在等情况进行异常处理。 |
| NFR-010 | 创建订单时应确保订单与订单明细同时保存。             |
| NFR-011 | AI 接口不可用时，系统应提供备用推荐或备用建议。        |
| NFR-012 | 客户端网络请求失败时应提示用户，而不是程序崩溃。         |

### 10.4 可维护性需求

| 编号      | 需求                                                  |
| ------- | --------------------------------------------------- |
| NFR-013 | 后端应按 Controllers、Models、Data、Services、DTOs 等目录划分代码。 |
| NFR-014 | 客户端 API 调用逻辑应封装到 ApiHelper 或 Service 类中。            |
| NFR-015 | 数据模型字段命名应统一，避免同一含义多种命名。                             |
| NFR-016 | 接口变更后应同步更新 API 文档。                                  |

### 10.5 安全性需求

| 编号      | 需求                      |
| ------- | ----------------------- |
| NFR-017 | 数据库连接字符串不应硬编码在代码中。      |
| NFR-018 | 后端应对输入数据进行校验。           |
| NFR-019 | EF Core 查询应避免 SQL 注入风险。 |
| NFR-020 | 生产环境不应开放过宽的 CORS 策略。    |

## 14. 风险分析与应对措施

| 风险                | 可能性 | 影响  | 应对措施                              |
