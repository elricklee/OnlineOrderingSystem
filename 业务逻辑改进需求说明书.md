# 线上点餐系统业务逻辑改进需求说明书

## 1. 文档目的

本文档用于指导当前项目进行一次面向答辩和后续完善的业务逻辑重构，目标是：

1. 找出当前系统中已经存在的业务逻辑缺陷与数据库设计问题。
2. 明确“菜品删除应为隐藏而非物理删除”是否已实现，以及实现是否完整。
3. 评估数据库表设计、表间关联和订单历史保留策略是否合理。
4. 给出一份可直接执行的改进需求说明书，便于在后续 Codex 对话中按项落地。

---

## 2. 结论摘要

### 2.1 菜品删除是否已按“隐藏”实现

结论：**基本做到了，但没有完全做严谨。**

当前实现中：

1. `Dish` 实体包含 `IsDeleted` 字段。
2. 管理端删除菜品调用的是 `DELETE /api/dishes/{id}`，后端实际执行的是逻辑删除。
3. 普通菜品查询会过滤 `IsDeleted = true` 的记录。

但当前实现仍有以下问题：

1. 后端依然暴露了 `/api/dishes/{id}/hard` 物理删除接口，不符合“只隐藏、不真删”的严格要求。
2. 没有“已隐藏菜品列表 / 回收站 / 恢复入口”的完整管理闭环。
3. 没有记录 `DeletedAt`、`DeletedBy`、`DeleteReason`，审计能力不足。
4. 没有使用 EF Core 全局过滤器，逻辑删除依赖开发者手工在每个查询里写条件，后续极易漏掉。

### 2.2 数据库设计是否合理

结论：**能跑通基础演示，但离“严谨、可扩展、业务闭环完整”的标准还有明显差距。**

优点：

1. `Orders` 与 `OrderItems` 已拆表。
2. `OrderItems` 保存了菜品名和下单时价格，具备订单历史快照意识。
3. `Orders` 同时保留了 `DiningTableId` / `TableNumber`、`DeliveryZoneId` / `DeliveryRegion`，说明已经考虑了历史快照。
4. `Users`、`DeliveryZones`、`DiningTables` 已有基础主表。

主要问题：

1. 餐桌占用逻辑设计不自洽，`IsOccupied`、`Status`、`CurrentOccupiedSeats` 三套状态并存但未统一。
2. `OrderItems.DishId` 没有配置正式外键，数据完整性不够强。
3. `DeliveryZone`、`DiningTable` 仍是物理删除，不利于保留历史和防误删。
4. 缺少“桌次/就餐会话”实体，导致一个餐桌多个订单、多个顾客、加单场景处理不严谨。
5. 枚举类状态大量使用字符串硬编码，缺少约束和字典化管理。
6. 数据库初始化同时混用 `EnsureCreated`、Migration、手工 SQL 补列，存在结构漂移风险。

### 2.3 表与表之间的关联是否合理

结论：**部分合理，但关键业务关联不够严谨。**

合理之处：

1. `Order` -> `OrderItems` 一对多是合理的。
2. `Order` -> `User`、`DiningTable`、`DeliveryZone` 采用可空关联并辅以快照字段，方向是对的。

不合理之处：

1. `OrderItems` 缺少到 `Dishes` 的正式外键约束。
2. 堂食订单直接关联 `DiningTable`，但没有“桌次/台次会话”中间层，导致加单、拼桌、多人订单、同桌多单逻辑难以保证正确。
3. 餐桌当前占用人数不应由订单删除/完成逻辑手工拼凑维护，应该由桌次会话统一驱动。

## 3. 当前代码审查发现的问题

以下问题基于当前代码实现审查得出。

### 3.1 菜品相关

#### P0-1 菜品虽然支持逻辑删除，但仍保留物理删除入口

风险：

1. 任何人只要知道接口就可以直接物理删除菜品。
2. 一旦物理删除被调用，将破坏“历史可追溯、删除可恢复”的教师要求。

现状：

1. `DELETE /api/dishes/{id}` 走逻辑删除。
2. `/api/dishes/{id}/hard` 仍可物理删除。

整改要求：

1. 课程版正式需求中取消对外暴露的物理删除接口。
2. 如确需保留，仅允许数据库维护脚本或超级管理员离线使用，不在正常 API 和 UI 中出现。

#### P1-1 菜品逻辑删除缺少回收站与审计字段

风险：

1. 管理端删除后无法查看已隐藏菜品。
2. 无法恢复误删。
3. 无法说明是谁、何时删除了菜品。

整改要求：

1. `Dishes` 增加 `DeletedAt`、`DeletedByUserId`、`DeleteReason`。
2. 管理端增加“已隐藏菜品”页签。
3. 支持恢复和再次上架。

#### P1-2 逻辑删除过滤依赖手工 `Where(!IsDeleted)`

风险：

1. 新增查询时非常容易漏写过滤条件。
2. 一旦漏写，隐藏菜品就会重新暴露给前端或统计模块。

整改要求：

1. 使用 EF Core 全局查询过滤器统一处理 `IsDeleted`。
2. 管理端回收站等少量场景显式使用 `IgnoreQueryFilters()`。

### 3.2 订单与堂食业务相关

#### P0-2 餐桌状态字段设计冲突，删除/完成/恢复后容易出现脏状态

风险：

1. `DiningTables` 同时存在 `IsOccupied`、`Status`、`CurrentOccupiedSeats` 三套状态源。
2. 订单确认时只改 `Status`，不改 `IsOccupied`。
3. 餐桌删除时却判断 `IsOccupied`，导致可能删除“实际上正被占用”的餐桌。
4. 订单逻辑删除、物理删除、恢复订单都没有正确回补或重建占座人数。

典型后果：

1. 餐桌界面显示“可用”，但剩余人数不对。
2. 已有顾客占座的餐桌仍可能被误删。
3. 恢复订单后，餐桌占用状态不回滚。

整改要求：

1. 废弃 `IsOccupied` 或将其改为由统一逻辑维护的派生状态。
2. 餐桌可用性以“桌次会话状态”作为唯一事实来源。
3. 删除订单、取消订单、完成订单、恢复订单都必须经过统一的桌次服务处理。

#### P0-3 缺少“桌次/台次会话”实体，堂食多单逻辑不严谨

风险：

1. 当前用 `UserId + DiningTableId` 近似识别“同一桌的同一批顾客”，逻辑不可靠。
2. 对游客用户（`UserId = null`）尤其容易误判，多个不同游客可能被当成同一人。
3. 加单、拼桌、分单、并单等场景没有严格业务载体。

整改要求：

新增 `TableSessions` 表：

1. 一张餐桌在某个时间段只会有一个有效桌次会话，或根据业务允许多个子订单挂在同一桌次下。
2. 订单关联 `TableSessionId`，不再直接用“用户是否相同”判断是否属于同一桌次。
3. 桌次会话维护 `PartySize`、`OpenedAt`、`ClosedAt`、`Status`、`OpenedBy`。

#### P1-3 订单删除语义不清

风险：

1. 当前“删除订单”实际上是逻辑删除，但它会影响堂食桌状态。
2. 订单作为经营历史数据，本不应被轻易删除。
3. 删除与取消在业务上语义混乱。

整改要求：

1. 正式业务中取消“删除订单”按钮。
2. 订单只能取消，不应从业务界面删除。
3. 如需隐藏异常单，使用“归档”或“作废”状态，而非普通删除。

#### P1-4 订单状态有状态机雏形，但缺少状态历史表

风险：

1. 只能看到当前状态，看不到流转轨迹。
2. 无法审计谁在何时改了状态。
3. 无法支撑后续骑手接单、催单、超时统计。

整改要求：

新增 `OrderStatusHistories` 表，至少记录：

1. `OrderId`
2. `FromStatus`
3. `ToStatus`
4. `OperatorUserId`
5. `OperatorRole`
6. `Remark`
7. `CreatedAt`

#### P1-5 设计文档包含骑手角色，但系统未真正落地

风险：

1. 设计文档与实现不一致。
2. 外卖订单“配送中”状态没有责任主体。

整改要求：

1. 若保留骑手角色，则增加骑手用户、配送指派、配送员信息维护。
2. 若课程范围不做骑手端，则在文档里明确“配送由管理员代操作”，避免答辩时前后矛盾。

### 3.3 配送区域与用户相关

#### P1-6 配送区域当前是物理删除，不适合长期业务

风险：

1. 已有历史订单的配送区域仍可能被直接删掉。
2. 后续区域误删后无法恢复。

整改要求：

1. 配送区域改为“停用/隐藏”，不允许普通物理删除。
2. 历史订单继续保留 `DeliveryZoneId` + `DeliveryRegionSnapshot`。

#### P1-7 用户有角色字段，但系统没有权限控制

风险：

1. 所有 API 默认裸奔。
2. 顾客理论上可以直接调用管理接口。
3. `Role` 字段目前只存不用。

整改要求：

1. 课程最小版至少区分“顾客端接口”和“管理端接口”。
2. 增强版接入 JWT 或 Session 鉴权。
3. 管理接口加授权校验。

#### P2-1 密码加密方案过于简单

风险：

1. 当前使用固定盐 SHA256，不适合正式生产。
2. 无每用户随机盐，抗撞库能力弱。

整改要求：

1. 改用 `PasswordHasher`、BCrypt 或 ASP.NET Core Identity。
2. 补充密码重置、禁用登录、最近登录时间等管理能力。

### 3.4 数据层相关

#### P0-4 数据库初始化策略混乱，存在结构不一致风险

风险：

1. 同时使用 `EnsureCreated()`、Migration、手写 `CREATE TABLE`、手写 `ALTER TABLE`。
2. `DbSeeder` 只补了部分列，没有完整覆盖当前模型。
3. 一旦本地数据库不是从空开始创建，就可能出现实体类与数据库结构不一致。

整改要求：

1. 正式环境只保留 Migration 方案。
2. 删除 `EnsureCreated()`。
3. `DbSeeder` 只保留数据初始化，不再负责结构修补。

#### P1-8 `OrderItems` 与 `Dishes` 缺少正式外键约束

风险：

1. `OrderItems.DishId` 可以写入不存在的菜品 ID。
2. 数据完整性依赖应用层判断，不够稳。

整改要求：

1. `OrderItems.DishId` 改为可空外键或保留外键但禁止物理删菜。
2. 推荐方案：保留 `DishId` 作为可空外键，同时保留 `DishNameSnapshot`、`UnitPriceSnapshot`。

## 4. 对当前数据库设计的总体评价

## 4.1 可保留的设计

以下设计建议保留：

1. `Orders` 与 `OrderItems` 分表。
2. `OrderItems` 保存快照字段。
3. `Users`、`DeliveryZones`、`DiningTables` 使用独立主表。
4. `Orders` 中保留 `DeliveryFee`，避免区域后续改价影响历史单。

## 4.2 需要调整的设计

以下设计建议调整：

1. `Dishes.Category` 不建议长期直接用字符串，建议独立分类表。
2. `DiningTables` 不应同时维护多个互相冲突的占用字段。
3. `Order.Status`、`OrderType`、`DiningTable.Status` 需要枚举化或字典化。
4. `DeliveryZones` 和 `DiningTables` 需要支持停用/隐藏，而不是直接物理删。
5. 需要新增桌次会话与订单状态历史表。

## 5. 目标业务规则设计

## 5.1 菜品管理业务规则

### 5.1.1 菜品状态

建议将菜品状态拆分为两个维度：

1. `IsDeleted`：是否已隐藏/归档。
2. `SaleStatus`：是否可售，可取值为 `OnSale`、`OffSale`、`OutOfStock`。

规则：

1. 顾客端只展示 `IsDeleted = false` 且 `SaleStatus = OnSale` 的菜品。
2. 管理端展示 `IsDeleted = false` 的全部菜品，包括可售和不可售。
3. 已隐藏菜品只能在回收站查看。
4. 已产生历史订单的菜品禁止物理删除。

### 5.1.2 菜品删除规则

1. 管理员点击删除时，不删除记录，只设置隐藏标记。
2. 删除后菜品不再出现在顾客端与普通管理列表。
3. 已隐藏菜品可以恢复。
4. 只有无历史订单且超级管理员离线维护时，才允许真正物理清理。

### 5.1.3 菜品字段建议

`Dishes` 建议至少包含：

1. `Id`
2. `Name`
3. `CategoryId`
4. `Price`
5. `Description`
6. `ImagePath`
7. `SpicyLevel`
8. `SaleStatus`
9. `SortOrder`
10. `CreatedAt`
11. `UpdatedAt`
12. `IsDeleted`
13. `DeletedAt`
14. `DeletedByUserId`
15. `DeleteReason`

新增 `DishCategories`：

1. `Id`
2. `Name`
3. `SortOrder`
4. `IsEnabled`
5. `IsDeleted`

## 5.2 堂食业务规则

### 5.2.1 引入桌次会话

新增 `TableSessions` 后，堂食应按以下规则运行：

1. 顾客选择餐桌后，系统创建桌次会话。
2. 同一餐桌下的多个订单都挂到同一个 `TableSessionId`。
3. 加单、分单、并单都围绕桌次会话处理。
4. 餐桌状态由桌次会话驱动，而不是由某个订单单独驱动。

### 5.2.2 餐桌状态机

建议统一为：

1. `Available`
2. `Reserved`
3. `Occupied`
4. `Cleaning`
5. `Disabled`

规则：

1. 有有效桌次会话时，餐桌为 `Occupied`。
2. 桌次全部完成结账后，餐桌转 `Cleaning` 或 `Available`。
3. 停用的餐桌为 `Disabled`，顾客端不可选。
4. 一个桌次可包含多个订单，但餐桌整体状态保持占用直到最后一个有效订单关闭。

### 5.2.3 餐桌人数规则

1. 餐桌的当前占用人数应来自桌次会话，不应由订单删除逻辑手工扣减。
2. 允许大桌多单，但总人数不得超过座位数。
3. 游客用户与注册用户都不应通过 `UserId` 决定是否属于同一桌次。

## 5.3 外卖业务规则

### 5.3.1 配送区域

1. 配送区域支持新增、修改、停用、恢复。
2. 不允许普通管理员物理删除已有历史订单关联区域。
3. 订单提交后，配送费和区域名称以订单快照为准，不受后续区域改价影响。

### 5.3.2 外卖订单状态

建议状态：

1. `Pending`
2. `Confirmed`
3. `Preparing`
4. `Ready`
5. `Delivering`
6. `Completed`
7. `Cancelled`

可选增强：

1. `Assigned`：已分配骑手但未取餐
2. `PickedUp`：骑手已取餐

## 5.4 订单业务规则

### 5.4.1 订单删除与取消

正式业务建议：

1. 订单不可直接删除。
2. 订单只能取消，取消必须记录原因。
3. 若需隐藏异常测试单，使用“作废/归档”而不是普通删除。

### 5.4.2 订单快照

订单必须保存以下快照，防止主数据变更影响历史：

1. 顾客姓名
2. 手机号
3. 收货地址
4. 配送区域名称
5. 配送费
6. 餐桌号
7. 菜品名称
8. 菜品单价
9. 菜品分类名称（建议）

### 5.4.3 订单状态历史

1. 每次状态变化都写入历史表。
2. 记录操作人、操作时间、原因、备注。
3. 管理端支持查看状态轨迹。

## 5.5 用户与权限规则

### 5.5.1 最低要求

1. 顾客仅可访问顾客端接口。
2. 管理员才可访问菜品管理、订单管理、用户管理、统计接口。

### 5.5.2 增强要求

角色建议：

1. `Customer`
2. `Admin`
3. `Kitchen`
4. `Rider`
5. `SuperAdmin`

不同角色的权限边界需要在接口层强制控制。

## 7. 接口改造需求

## 7.1 菜品接口

### 保留

1. `GET /api/dishes`
2. `GET /api/dishes/{id}`
3. `POST /api/dishes`
4. `PUT /api/dishes/{id}`
5. `DELETE /api/dishes/{id}`

### 新增

1. `GET /api/dishes/recycle-bin`
2. `PUT /api/dishes/{id}/restore`
3. `PUT /api/dishes/{id}/sale-status`

### 废弃

1. `DELETE /api/dishes/{id}/hard`

## 7.2 订单接口

### 建议调整

## 9. 验收标准

## 9.1 菜品删除验收

1. 管理员删除菜品后，数据库记录仍存在。
2. 该菜品不再出现在顾客端与普通管理列表。
3. 管理端回收站可看到该菜品。
4. 可以恢复该菜品。
5. 历史订单中的菜品名称、价格、统计数据不受影响。
6. 无任何业务 UI 可以直接物理删除菜品。

## 9.2 堂食餐桌验收

1. 一个餐桌可被正确标识为可用、占用、清洁中、停用等状态。
2. 同桌加单不会重复错误占座。
3. 不同顾客不会因 `UserId = null` 被误判为同一桌次。
4. 订单取消、完成、恢复后，餐桌状态与占座人数始终正确。
5. 被占用的餐桌不可被删除或停用。

## 9.3 配送区域验收

1. 已有历史订单的配送区域不能被普通流程物理删除。
2. 区域停用后顾客端不可选，但历史订单正常显示。

## 9.4 订单验收

1. 订单状态只能按允许的状态机流转。
2. 每一次状态变化都有历史记录。
3. 取消订单必须填写原因。
4. 管理端能查看状态轨迹。

## 9.5 数据层验收

1. 数据库结构仅通过 Migration 演进。
2. 不再使用 `EnsureCreated + 手工补列` 混合方案。
3. 逻辑删除实体默认自动过滤。
4. `OrderItems.DishId` 具备明确的数据完整性策略。

## 12. 参考依据

### 12.1 项目内文档

1. 需求规格说明书已明确：顾客端默认只展示可售菜品，删除菜品时已有历史订单应优先“设为不可售”而非物理删除。
2. 软件设计文档已明确：系统包含堂食、外卖、订单状态流转、餐桌、配送区域、用户角色等核心业务。

### 12.2 业界参考

1. Toast、Billfold 等餐饮系统普遍采用“归档/隐藏菜品”而不是直接删除有销售历史的菜品。
2. 多个餐饮桌台管理系统采用明确的桌台状态机，并保证桌台在全部有效订单完成前持续保持占用。
3. EF Core 官方推荐使用 Global Query Filter 处理软删除，以避免每个查询手写过滤条件。
